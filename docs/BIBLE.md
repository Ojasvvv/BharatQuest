## 1. WHAT THIS PROJECT IS, IN PLAIN ENGLISH

Apatheia solves the problem of letting AI agents write and immediately test code without accidentally blowing up your server or silently eating all your cloud budget. For developers building autonomous AI agents, it is extremely dangerous to take Python or JavaScript generated by an AI and just run it natively on a server—the AI might hallucinate an infinite loop, try to read sensitive local files, or call malicious external APIs. 

Technically, Apatheia is a sub-millisecond WebAssembly (WASM) code execution sandbox built in Rust. It securely executes untrusted, dynamically generated JavaScript and Python code by running interpreters inside memory-isolated, heavily metered WASM instances. It guarantees safety through strict opcode instruction limits (fuel) and wall-clock timeouts, preventing infinite loops and runaway resource consumption. 

## 2. THE CORE ARCHITECTURE

WebAssembly (WASM) is a binary instruction format originally designed to let high-performance code run safely inside web browsers. It achieves this by giving the code its own strictly bounded block of "linear memory" that it cannot escape. WASI (WebAssembly System Interface) is the standard that brings WASM outside the browser to servers, providing controlled, capability-based access to the operating system (like file descriptors and clocks). The `wasm32-wasi` compile target produces WASM binaries that expect a WASI-compliant host environment. By running untrusted code inside WASM, we gain absolute memory isolation—the guest code literally cannot address memory outside its sandbox sandbox.

### Why Compile Interpreters to WASM?
We compile QuickJS and MicroPython into WASM rather than compiling the AI's generated code directly into WASM. If we tried to compile the AI's dynamically generated Python script to WASM, we'd have to invoke a heavy compiler toolchain on every single request, taking hundreds of milliseconds or seconds. Instead, we compile the language *interpreters* once into WASM modules (`quickjs.wasm` and `micropython-wasi.wasm`). On every request, we just spin up a fresh WASM sandbox running the interpreter, and pass the AI's string to it. (See DECISIONS.md for a deep dive on this trade-off).

### Wasmtime Pooling & InstancePre
Wasmtime is the Rust-based runtime that executes our WASM binaries. Normally, spinning up a WASM module takes a few hundred microseconds to parse and compile the WebAssembly into native machine code. 
Wasmtime provides an `InstancePre` type, which is a pre-compiled snapshot of the WASM module. Using Wasmtime's Pooling Allocator, we create instances via Copy-on-Write (CoW) memory mapping. An "instance" is an active execution environment of the WASM module with its own linear memory. "Cloning" at the OS level means using `mmap` to map the exact same physical memory pages of the `InstancePre` snapshot to a new virtual address space. Because it's CoW, the memory is shared instantly until the instance modifies a byte, at which point the OS lazily copies just that page. This brings initialization time down to ~20-50 microseconds.

### Request Lifecycle
```text
[API] req arrives
  │
[Engine] tokio::task::spawn_blocking wrapper
  │
  ├─► [Engine] Wasmtime creates fresh Instance from InstancePre (clone)
  │
  ├─► [Engine] write JS into linear memory via alloc_buffer()
  │
  ├─► [Engine] eval_js() executes QuickJS synchronously 
  │
  ├─► [Engine] read stdout/stderr from linear memory
  │
[Engine] Instance dropped immediately (memory discarded instantly)
  │
[API] respond with JSON
```

## 3. CRATE-BY-CRATE WALKTHROUGH

Apatheia is structured as a Cargo Workspace to enforce strict boundary separation. 

### `api`
Provides the Axum REST server and WebSocket interfaces. 
- **`main.rs`**: Entry point. Sets up the tokio runtime and `AppState`.
- **`handlers.rs`**: Contains `execute_handler()` for HTTP POST routing, and `stream_metrics_handler()` for the WebSocket. 
- **`models.rs`**: Serde schema definitions for inputs and outputs.
- **`state.rs`**: `AppState` holds `Arc<RuntimePool>` and tracking maps for `max_iterations`.

### `engine`
The core sandbox orchestrator interacting with `wasmtime`.
- **`lib.rs`**: Exposes `RuntimePool` and `ExecutionResult`.
- **`pool.rs`**: Initializes two separate `wasmtime::Engine` environments—one with the Pooling Allocator for QuickJS (Reactor module), and one without for MicroPython (Command module requiring exceptions).
- **`runtime.rs`**: The heavy lifter. Defines `RuntimeHandle::execute()`. Uses `tokio::task::spawn_blocking` to execute WASM without blocking the async executor.
- **`sandbox.rs`**: Defines constants and fallback configs. 
- **`error.rs`**: `EngineError` defining `FuelExhausted`, `WallClockTimeout`, `MemoryMarshal`.

### `telemetry`
Shared observability layer.
- **`lib.rs`**: Defines `ExecutionMetrics`, capturing `instance_clone_time_us`, `execution_time_us`, `memory_marshal_us`, `total_time_us`, and `fuel_consumed`. 

### `quickjs-wasm`
Crate responsible for invoking the C compiler toolchain via `build.rs` or `build.sh` to generate the WASM interpreter. 
- **`build.sh`**: The shell script that invokes `clang` with WASI sysroot.
- **`wrapper.c`**: The FFI bridge inside the WASM binary exposing `alloc_buffer`, `eval_js`, etc.

### `ffi-bridge`
Intended to provide the SSRF-firewalled host function for network requests. Currently stubbed out with only an `error.rs` definition.

### `dashboard`
The React + Vite front-end. (See Chapter 10).

## 4. THE QUICKJS-TO-WASM BUILD PIPELINE

The `quickjs-wasm/build.sh` script cross-compiles QuickJS to WASM. 
It uses **wasi-sdk-25.0**, a clang-based toolchain specifically bundled with a WebAssembly sysroot (a `libc` implementation based on `wasi-libc`).

**Important build flags**:
- `--target=wasm32-wasi`: Tells LLVM to emit WASM instructions and WASI imports.
- `-mexec-model=reactor`: Crucial. Without this, clang builds a "Command" module that expects to run `main()` and immediately exit (`_start`). Reactor models export `_initialize` (running global C constructors once) and remain alive so we can repeatedly call custom FFI functions.
- `-Wno-*`: Silences noisy compiler warnings.
- `-mthread-model single`: WASI preview 1 doesn't support threads. This drops atomics.
- `-Wl,--export=...`: Instructs the WASM linker to explicitly export `alloc_buffer`, `eval_js`, and `--export-memory`.

**FFI Exports (in `wrapper.c`)**:
- `alloc_buffer(size_t len)`: Calls `malloc` inside the WASM linear memory and returns a pointer integer to the host.
- `eval_js(void* ptr, size_t len)`: Takes the string at the pointer, feeds it to `JS_Eval()`, captures `stdout`, and returns `0` on success, `1` on Parse Error, and `2` on Runtime Error.

## 5. FUEL METERING AND THE WATCHDOG

"Fuel" in Wasmtime is a deterministic, instruction-based accounting system. When Wasmtime compiles WASM into native machine code, it injects extra instructions that decrement a counter based on the cost of the WASM opcodes being executed. 
Fuel is necessary because WASM execution is purely deterministic CPU-bound work. If an AI generates `while(true){}`, the WASM loop will spin forever.

However, fuel alone is insufficient. If a WASM guest calls a host function (like a synchronous file read or an HTTP fetch) that blocks for 30 seconds, it consumes *zero* CPU fuel during that sleep, but the server thread is completely locked. 
To fix this, Apatheia layers a **Wall-clock watchdog** on top.
Code path: `engine/src/runtime.rs:269`
```rust
tokio::time::timeout(Duration::from_millis(timeout_ms), execute_future).await
```
If fuel runs out, the Wasmtime instance traps immediately and throws `Trap::OutOfFuel`. 
If the execution hangs on a low-fuel blocking operation, `tokio` cancels the spawned blocking task after `timeout_ms`. The OS thread is killed/dropped and the instance is destroyed.

## 6. THE API CONTRACT, IN FULL

The API runs over HTTP POST `/v1/execute` and WS `/v1/stream`.

**ExecuteRequest**
```json
{
  "request_id": "f-py-01",
  "language": "python",
  "code": "print(42)",
  "timeout_ms": 5000,
  "memory_limit_mb": 64,
  "parent_request_id": "f-retry-root"
}
```

**ExecuteResponse Variants**

1. Success
```json
{
  "status": "success",
  "request_id": "f-py-01",
  "language": "python",
  "stdout": "42\n",
  "metrics": {
    "instance_clone_time_us": 66,
    "execution_time_us": 1330,
    "memory_marshal_us": 0,
    "total_time_us": 1479,
    "fuel_consumed": 124910
  }
}
```

2. Runtime Error (Caught exception fed back to LLM)
```json
{
  "status": "runtime_error",
  "request_id": "f-py-02",
  "language": "python",
  "metrics": { "instance_clone_time_us": 54, "execution_time_us": 1647, "memory_marshal_us": 0, "total_time_us": 1783, "fuel_consumed": 168100 },
  "error_telemetry": {
    "type": "AttributeError",
    "trace": "Traceback...\nAttributeError: 'list' object has no attribute 'sumAll'\n",
    "message": "'list' object has no attribute 'sumAll'"
  },
  "llm_feedback_prompt": {
    "role": "system",
    "content": "Execution failed: AttributeError: 'list' object has no attribute 'sumAll'. Review your code and provide a corrected script."
  }
}
```

3. Rejected (Fuel, Timeout, Memory limit hit)
```json
{
  "status": "rejected",
  "request_id": "f-py-03",
  "language": "python",
  "reason": "out_of_fuel",
  "metrics": { "instance_clone_time_us": 64, "execution_time_us": 4014, "memory_marshal_us": 0, "total_time_us": 4183, "fuel_consumed": 50000000 }
}
```

The system also returns `429 Too Many Requests` if the `parent_request_id` has failed more than `MAX_ITERATIONS` times.

## 7. THE SSRF FIREWALL

Server-Side Request Forgery (SSRF) is an attack where an attacker asks the server to fetch a URL that points to an internal, private network (like AWS metadata endpoint `169.254.169.254`). 
A naive blocklist on the URL string fails due to DNS rebinding (the attacker points `evil.com` to `8.8.8.8`, bypassing the string check, then changes the DNS to `127.0.0.1` immediately before the server actually connects).

> **GAP IDENTIFIED**: Phase 4 FFI Bridge is pending. The actual `resolve -> validate -> connect` loop logic does not exist in the codebase. Only the `error.rs` file exists in `ffi-bridge`. Outbound network capabilities for the AI sandbox are currently entirely un-implemented.

## 8. THE ASYNC/SYNC BRIDGE PROBLEM

QuickJS eval functions are standard synchronous C code. When we call them via Wasmtime's FFI, the Rust thread executing them is completely blocked until the JavaScript finishes.
In an asynchronous Rust framework like `tokio`, blocking a worker thread for an extended period is disastrous—it starves the executor and crashes the entire web server's throughput.
**Implementation Chosen**: We implemented `tokio::task::spawn_blocking` (in `engine/src/runtime.rs:104`). This moves the synchronous WASM execution off the core async event loop and onto a dedicated blocking thread pool provided by `tokio`. We did NOT implement the heavier Promise/microtask pumping architecture.

## 9. TELEMETRY AND METRICS

The 5 tracked metrics are measured sequentially to profile bottlenecks:
- `instance_clone_time_us`: Starts before `instantiate()` and stops after `_initialize()`. Measures the pure overhead of the Wasmtime pooling allocator.
- `memory_marshal_us`: Starts before `alloc_buffer`, stops after writing to WASM memory. Measures host-to-guest data transfer overhead. (Always 0 for Python, as it uses WASI `argv` instead of FFI pointers).
- `execution_time_us`: Starts right before calling `eval_js` / `_start` and ends immediately after. This is the raw interpreter performance.
- `fuel_consumed`: Final remaining fuel subtracted from initial budget.
- `total_time_us`: Calculated around the entire `spawn_blocking` task boundary. 

## 10. THE DASHBOARD

The dashboard uses React + Vite + Tailwind CSS.
- **TopBar.tsx**: Contains dynamic WebSocket connection status (connected/reconnecting/disconnected).
- **Waterfall.tsx**: Renders the `events` stream from the WS connection, visualizing real-time metrics with green/red/orange bars.
- **MetricRow.tsx**: Summarizes p50 latency metrics.
- **AgentSimulator.tsx**: Sends dynamic POST requests to `/v1/execute`. It implements a visual trace of the "Self-healing loop", simulating how an LLM would use the `llm_feedback_prompt` response. It includes a multi-sandbox "Compare Mode".

## 12. TESTING

Tests reside heavily in `api/src/handlers_test.rs` and `engine/tests/python_integration.rs`.
- `test_python_sandbox_isolation`: Proves that two sequential executions of Python cannot see each other's global variables or states.
- `test_infinite_loop_fuel_trap_rejection`: Proves that a `while(true)` JS script correctly trips the Wasmtime fuel exhaustion limit and returns `out_of_fuel`.
- `test_max_iterations`: Proves the self-healing rate limit tracks `parent_request_id` properly.
- *Gap identified*: No FFI or network tests exist because the SSRF logic is descoped.

## 13. KNOWN LIMITATIONS AND NON-GOALS

- Network fetch is fully unimplemented (`unimplemented!` practically missing from the codebase entirely).
- Multithreading inside WASM is completely disabled.
- Only standard libraries are available in Python (MicroPython subset). No `numpy` or `pandas`.

## 14. GLOSSARY

- **CoW (Copy-on-Write)**: An operating system memory optimization. When copying a large chunk of memory, the OS doesn't physically duplicate the RAM. Both copies share the same memory until one tries to modify it, at which point a physical copy is lazily made.
- **FFI (Foreign Function Interface)**: The mechanism by which a program written in one language can call routines written in another (e.g., Rust calling C functions exported by WebAssembly).
- **InstancePre**: Wasmtime's term for a compiled WebAssembly module that has been pre-processed into executable native machine code, waiting to be instantiated.
- **SSRF (Server-Side Request Forgery)**: A vulnerability where an attacker manipulates a server into making HTTP requests to internal, private network endpoints.
- **WASI (WebAssembly System Interface)**: A standardized API allowing WebAssembly code to securely interact with the host operating system (files, clocks, random numbers) outside of a browser.
- **WASM (WebAssembly)**: A safe, portable, low-level binary instruction format.
